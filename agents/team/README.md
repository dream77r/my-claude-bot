# Team Hub — командный Telegram-ассистент

Агент `team` — generic-ассистент для группового чата в Telegram. Он:

- Тихо читает всё в разрешённом топике, отвечает только на упоминания и reply.
- Ведёт базу знаний команды (участники, задачи, решения, база компании).
- Делает утренние стендапы и вечерние дайджесты по расписанию.
- Принимает от участников напоминания вида «напомни мне в HH:MM про X».
- Трекает обещания (план/факт, переносы, follow-up).
- Прогоняет Dream раз в 3 часа: извлекает факты и обновляет wiki с перекрёстными ссылками.
- Пишет ночное саммари дня.

## Generic-архитектура

Вся специфика конкретной команды (chat_id, таймзона, расписание, участники,
база знаний о компании) хранится в файлах памяти, а не в коде. Это
позволяет развернуть бота на любом сервере и настроить под конкретную
команду **через Telegram**, без правки YAML или Python.

## Быстрый старт (для self-hosters)

1. **Подними бота** по общей инструкции my-claude-bot. Убедись, что
   `TEAM_BOT_TOKEN` задан в `.env`.
2. **Пригласи team в нужный групповой чат Telegram.** Дай ему права на
   чтение сообщений.
3. **Открой DM с master-агентом `me`.** Напиши ему: _«Настрой team в группе
   моей команды»_.
4. Master задаст пошаговые вопросы (chat_id, топик, таймзона, расписание,
   участники, база знаний) и запишет всё в
   `memory/groups/{chat_id}/settings.json`.
5. Через 5 минут первый `schedule_check` подхватит новую группу и начнёт
   работать согласно расписанию.

Детальный сценарий онбординга — в `agents/me/skills/setup-team.md`.

## Сандбокс

Агент `team` — `role: worker`, все операции с ФС ограничены `agents/team/`.
В Telegram — одним топиком в одной группе через `allowed_topic` в
settings.json. Если нужно что-то наружу — делегирование мастеру `me` через
FleetBus.

## Расширение

Хочешь добавить новый ритуал (например, еженедельную ретроспективу по
пятницам)?

1. Добавь шаблон в `templates/weekly_retro.md`.
2. Допиши логику в `skills/daily-sync.md` (или создай новый скилл).
3. Обнови `templates/schedule_check.md` — добавь новый триггер.
4. Поля расписания — в `settings.json` → `daily_sync` или `weekly_rituals`.

Никаких правок Python-кода не требуется.

## Протокол доставки сообщений из cron и фоновых задач

Когда агент работает по запросу пользователя, ответ летит обратно в тот
же чат автоматически. Когда агент просыпается по cron или heartbeat,
«текущего пользователя» нет, и сообщение нужно явно адресовать.

Для этого используется папка `memory/dispatch/`. Агент пишет туда JSON-файл:

```json
{
  "chat_id": 123456789,
  "message_thread_id": 42,
  "text": "Текст сообщения",
  "parse_mode": "Markdown",
  "source": "morning_standup"
}
```

`src/dispatcher.py` раз в 5 секунд сканирует `dispatch/` у каждого
агента, публикует каждое сообщение в FleetBus как `MessageType.OUTBOUND`
с `metadata.message_thread_id`, а `telegram_bridge._send_via_bot`
передаёт его в нужный топик группы. Обработанные файлы удаляются,
сломанные — уходят в `dispatch/failed/` для диагностики.

Обязательные поля — `chat_id` и `text`. Остальные опциональны.

## Что явно не реализовано в этой архитектуре

- Парсинг относительных времён в русском языке для `reminders` («завтра в
  пятницу после обеда»). Скилл описывает правила; качество парсинга
  зависит от модели.
- UI для просмотра/редактирования настроек через inline-кнопки (сейчас
  только через диалог с master-агентом).
- Сервис уведомлений в Telegram о неудачных dispatch-файлах (они просто
  копятся в `dispatch/failed/`). При необходимости можно добавить
  отдельную cron-задачу, которая раз в сутки тегает владельца группы.

Эти места — кандидаты на следующую итерацию после первого живого теста.
