# Skill: vault-init — первичная настройка vault'а

## Когда активировать

- При первом сообщении пользователя, если `memory/.vault-config.json` отсутствует
  или содержит `"initialized": false`
- Когда пользователь явно просит настроить архив ("давай настроим", "инициализируй")
- Когда пользователь кидает архив (zip/папку) без предварительных инструкций —
  это значит, он хочет импортировать существующий архив

## Приоритет

**ВЫСОКИЙ.** Пока vault не инициализирован, другие скиллы (document-ingest,
archive-search, schema-evolve) не должны запускаться. Если пользователь пытается
загрузить документ или искать — мягко переведи разговор на vault-init.

## Три режима инициализации

Всегда предлагай в таком порядке приоритета: импорт → пример → с нуля.

### Режим A — Импорт существующего архива

**Когда:** пользователь кидает zip, папку, или несколько файлов сразу.

**Шаги:**

1. Если это zip — распакуй во временную папку. Если просто файлы — сложи во
   временную папку.
2. **Проанализируй структуру:**
   - Какие есть подпапки? Их названия?
   - Какие типы файлов (pdf, docx, md, json)?
   - Если есть markdown-файлы, прочитай несколько — есть ли YAML frontmatter?
     Какие поля?
   - Есть ли `SCHEMA.md`, `CLAUDE.md`, `_index.md`, `templates/` — то есть
     структурированная wiki-система?
   - Считаются ли wiki-ссылки `[[slug]]` внутри markdown?
3. **Сообщи пользователю результат анализа** в структурированной форме:
   ```
   Посмотрел твой архив. Вижу:
   - N папок: {список}
   - M файлов, из них K — PDF/DOCX (сырые документы), L — markdown (карточки)
   - {если есть frontmatter} Поля в frontmatter: {список}
   - {если есть wiki-ссылки} X wiki-ссылок, Y битых
   - Домен похож на: {предположение}
   ```
4. **Предложи схему** словами. Основа — то, что ты увидел в архиве, плюс
   потенциальные улучшения:
   - Предложи N сущностей (названия, что хранят)
   - Предложи таксономию тегов из существующих тегов
   - Предложи обязательные поля
   - Если видишь системные улучшения (например, не хватает сущности для группировки
     документов по проектам/точкам) — скажи об этом отдельным пунктом
5. **Итерируй с пользователем** — он может сказать "добавь ещё сущность X",
   "переименуй Y в Z", "убери тег W". Правь предложение, пока не получится "ОК".
6. **Примени:**
   - Создай `memory/SCHEMA.md` из `meta-templates/schema-skeleton.md` с
     подставленными значениями
   - Создай `memory/.vault-config.json` с полной таксономией и `initialized: true`
   - Создай `memory/wiki/{entity_type}/` для каждой сущности
   - Создай `memory/wiki/{entity_type}/_index.md` — пустые индексы
   - Создай `agents/archivist/templates/{entity_type}.md` из `meta-templates/entity-template.md`
     и `meta-templates/document-template.md`
   - Перенеси сырые файлы из импортированного архива в `memory/raw/`
   - Если в импортированном архиве были существующие markdown-карточки — перенеси
     их в соответствующие папки, обновляя формат при необходимости
7. **Зафиксируй в log.md:**
   ```
   - YYYY-MM-DD HH:MM — [schema] vault инициализирован из импорта,
     домен: {описание}, сущности: {список}, карточек перенесено: N
   ```
8. **Подтверди пользователю:** короткий отчёт — что создано, сколько файлов
   перенесено, что делать дальше.

### Режим B — Начать с примера

**Когда:** пользователь говорит "начни с нуля, покажи примеры" или явно выбирает
один из примеров.

**Шаги:**

1. **Покажи список доступных примеров** из `agents/archivist/examples/`:
   - Прочитай `examples/README.md`
   - Для каждого примера прочитай его `README.md` и дай краткое описание
2. **Спроси, какой подходит** или предложи самый близкий по описанию пользователя.
3. **Адаптируй пример** — спроси пользователя о конкретике:
   - Какие именно юрлица/проекты/сущности у тебя? (названия, идентификаторы)
   - Есть ли особенности, которых нет в примере?
4. **Скопируй базу из примера:**
   - `examples/{name}/SCHEMA.md` → `memory/SCHEMA.md`
   - `examples/{name}/vault-config.json` → `memory/.vault-config.json`
     (обновив `initialized: true`, `created`, `access.owner`)
   - `examples/{name}/templates/*.md` → `agents/archivist/templates/*.md`
5. **Создай пустые папки** для всех сущностей из схемы
6. **Зафиксируй в log.md**
7. **Подтверди пользователю**

### Режим C — С нуля через вопросы

**Когда:** у пользователя нет существующего архива, и ни один пример не подходит.

**Шаги:**

1. **Задай 3 главных вопроса одним сообщением** (чтобы не растягивать):
   ```
   Ок, с нуля. Чтобы собрать схему под тебя, мне нужно понять три вещи:

   1. Что за домен? Одно-два предложения: малый бизнес, юридическая практика,
      научная работа, личные документы, что-то другое?

   2. Какие типы документов будешь грузить? Примеры: договоры, счета, акты,
      паспорта, научные статьи, рецепты...

   3. Кто "действующие лица" в твоих документах? С чем документы связаны?
      Примеры: компании, люди, проекты, объекты недвижимости, события,
      пациенты...

   Отвечай как удобно — текстом, списком, как угодно.
   ```
2. **Проанализируй ответы:**
   - Преобразуй "действующие лица" в entities (например, "компании, люди,
     проекты" → `entities: [company, person, project]`)
   - Преобразуй "типы документов" в document_types
   - Определи главный домен
3. **Задай 1-3 уточняющих вопроса** (только важные, не перегружай):
   - Важны ли сроки/даты истечения? (если да — добавь lint на expiring)
   - По каким ключам дедуплицировать сущности? (ИНН для организаций,
     email для людей, что-то ещё?)
   - Работа одиночная или с командой? (влияет на access control)
4. **Сгенерируй схему** — покажи словами что будет создано
5. **Подтверди с пользователем**, дай шанс поправить
6. **Примени** — то же самое, что в режимах A и B: создать файлы схемы,
   конфига, папки сущностей, шаблоны
7. **Зафиксируй в log.md**
8. **Подтверди пользователю**

## Обязательные вопросы при любом режиме

Независимо от режима, в конце инициализации:

1. **Владелец (access.owner):** "Это твой основной Telegram ID? Я запомню тебя
   как владельца — только ты сможешь менять схему."
   - Записать в `.vault-config.json → access.owner`
2. **Editors (опционально):** "Будет ли кто-то ещё загружать документы? Например,
   бухгалтер или помощник? Если да — дай их Telegram username'ы или попроси их
   написать мне один раз, я запомню их ID."
   - Пока оставить `editors: []`, потом команда `/grant-editor`

## Что создаётся после инициализации

```
memory/
  CLAUDE.md              ← уже есть из memory_template, не трогаем
  SCHEMA.md              ← создан заново под домен
  .vault-config.json     ← заполнен полностью, initialized: true
  raw/                   ← пустая (или с импортированными файлами)
  wiki/
    _home.md             ← обновлён с ссылками на сущности домена
    log.md               ← первая запись "vault инициализирован"
    {entity_type_1}/
      _index.md
    {entity_type_2}/
      _index.md
    ...

agents/archivist/templates/
  {entity_type_1}.md
  {entity_type_2}.md
  ...
```

## Первый ответ пользователю после инициализации

Коротко и по делу:
```
Готово. Настроил под {домен}.

Сущности: {список} ({N} всего)
Типы документов: {список}
Теги: {краткий обзор}

{Если было импортировано} Перенёс {N} файлов в raw/, создал {M} карточек.

Что дальше:
- Кидай мне документы — сам разложу
- Спрашивай — найду по реквизитам, датам, связям
- Если что-то в схеме захочется поправить — скажи "добавь сущность X" или
  "переименуй Y"
```

## Запреты

- НЕ инициализируй vault без явного подтверждения пользователя. Если сомнения —
  покажи предлагаемую схему и спроси.
- НЕ копируй сырые файлы в `raw/` без согласия пользователя (если это импорт
  большого архива — спроси, хочешь всё или только подмножество).
- НЕ создавай сущности, которых пользователь не назвал. Добавлять — только
  через `schema-evolve` после инициализации.
- НЕ ломай `memory_template/`. Все правки — только в `memory/` конкретного
  инстанса.

## Формат ответа во время диалога

- Структурированный, но не бюрократичный
- Один вопрос или один набор вопросов за раз
- Показывай "что я вижу" перед "что я предлагаю"
- Давай пользователю контроль: всегда можно сказать "не так" и переделать
