# Skill: schema-evolve — изменение схемы по запросу

## Когда активировать

- Owner просит добавить/изменить/удалить что-то в схеме
- Owner принимает предложение от SchemaAdvisor
- Нужно расширить таксономию тегов, когда добавляется сущность и автоматически
  генерируются связанные теги

## Предусловия

1. Vault инициализирован
2. **Только Owner.** Editor и Reader получают вежливый отказ: "Изменение схемы
   доступно только владельцу архива."
3. Пользователь сформулировал конкретное изменение (а не "сделай лучше")

## Виды изменений, которые поддерживаются

### 1. Добавление новой сущности

**Запрос:** "добавь сущность X" / "нужна папка для Y"

**Шаги:**
1. Уточни обязательные поля ("какие поля у {X} должны быть обязательными?")
2. Уточни дедупликацию ("по какому полю проверять дубли? обычно это уникальный
   идентификатор — ИНН, email, номер паспорта")
3. Уточни связи с существующими сущностями ("с кем {X} связан? например,
   `outlet` связан с `legal_entity`")
4. Сгенерируй diff:
   ```
   Что изменится:
   - В .vault-config.json → entities добавлю "{X}"
   - Создам папку wiki/{x}/ и _index.md
   - Создам шаблон templates/{x}.md из meta-templates/entity-template.md
   - В .vault-config.json → required_fields_by_type добавлю {X}: {поля}
   - В .vault-config.json → dedup_keys добавлю {X}: [{ключ}]
   - В .vault-config.json → tags добавлю раздел автогенерации "{x}-*"
   - Обновлю SCHEMA.md, добавлю раздел "Сущности"
   ```
5. Показать пользователю, ждать подтверждения
6. Применить все изменения атомарно
7. Зафиксировать в log.md: `[schema] добавлена сущность {X}`
8. Предложить: "Хочешь пройтись по существующим документам и поискать упоминания
   {X}, чтобы создать первые карточки автоматически?"

### 2. Добавление подтипа документа

**Запрос:** "добавь тип документа X" / "у меня будут Y, обрабатывай их отдельно"

**Шаги:**
1. Уточни обязательные поля для этого подтипа
2. Уточни в какой папке хранить (обычно `wiki/documents/` или отдельная)
3. Уточни, связан ли он с родительским документом (например, акт сверки связан
   с договором)
4. Diff:
   ```
   Что изменится:
   - В .vault-config.json → document_types добавлю "{X}"
   - В .vault-config.json → required_fields_by_type добавлю {X}: {поля}
   - Создам шаблон templates/{x}.md из meta-templates/document-template.md
   - Обновлю SCHEMA.md
   ```
5. Подтверждение и применение

### 3. Добавление поля в существующую сущность

**Запрос:** "добавь поле X в контрагентов" / "у договоров должно быть Y"

**Шаги:**
1. Уточни — обязательное или опциональное
2. Если обязательное — предупреди: "Существующие карточки без этого поля
   станут невалидными при lint. Хочешь их обновить автоматически? Я пройдусь
   по каждой, попытаюсь извлечь {X} из связанных документов или попрошу тебя
   ввести вручную."
3. Diff (короткий — обычно только .vault-config.json и шаблон)
4. Подтверждение
5. Применение
6. Если обязательное — запустить проход по существующим карточкам

### 4. Добавление тега в таксономию

**Запрос:** "добавь тег X" / "у меня появилась категория Y"

**Шаги:**
1. Уточни в какой раздел таксономии (relation? category? payment?). Если раздела
   нет — предложи создать.
2. Diff (только `.vault-config.json → tags`)
3. Подтверждение и применение (обычно моментально, без сложных миграций)

### 5. Переименование сущности / поля / тега

**Запрос:** "переименуй X в Y"

**ОСТОРОЖНО.** Переименование ломает wiki-ссылки. Требуется полная миграция.

**Шаги:**
1. Предупреди: "Переименование затронет {N} карточек и {M} ссылок. Я обновлю
   все вхождения {X} → {Y}. Откатить потом будет сложно — убедись, что это то,
   что ты хочешь."
2. Подтверждение (явное, не просто "да")
3. Применение:
   - Переименовать папку / поле / тег в `.vault-config.json`
   - Пройтись Grep'ом по всем карточкам, заменить старое имя на новое
   - Если это slug файла — переименовать файл и обновить ссылки
   - Обновить SCHEMA.md
4. Запуск `vault-lint` для проверки целостности после миграции
5. Зафиксировать в log.md: `[schema] переименование {X} → {Y}, затронуто {N}
   карточек, {M} ссылок`

### 6. Удаление сущности / поля / тега

**ОЧЕНЬ ОСТОРОЖНО.** Удаление = потеря данных.

**Шаги:**
1. Предупреди: "Удаление сущности {X} удалит {N} карточек и {M} связей.
   Ты хочешь именно УДАЛИТЬ, или ARCHIVE (пометить как устаревшую, но
   сохранить)?"
2. Рекомендовать **archive**, а не delete, в большинстве случаев
3. Если всё-таки delete — требовать **явного многократного подтверждения**
4. Предложить экспортировать удаляемое в backup-файл перед удалением
5. После подтверждения — применить
6. Зафиксировать в log.md с явным указанием деструктивности

## Важные правила

1. **Каждое изменение — атомарно.** Если что-то идёт не так на шаге 3 из 5 —
   откатить шаги 1-2. Не оставлять vault в промежуточном состоянии.
2. **Каждое изменение — логируется.** log.md записывается перед применением,
   чтобы история была точной даже при ошибках.
3. **Каждое деструктивное действие — требует явного подтверждения.**
4. **SCHEMA.md и .vault-config.json синхронизируются.** Нельзя обновить одно
   без другого.
5. **Ссылки на несуществующие файлы не создаются.** После любой миграции —
   запустить `vault-lint` для проверки.

## Формат ответа

- Всегда начинай с "что будет сделано" (diff в текстовой форме)
- Всегда заканчивай конкретным вопросом подтверждения
- После применения — короткий отчёт, что сделано
- Не скрывай последствия — если что-то потенциально сломается, скажи

### Пример 1 — добавление сущности

```
User: добавь сущность склад, у меня появились склады

Архивариус: Ок, добавляю сущность "warehouse". Пара уточнений:

1. Обязательные поля: адрес — однозначно. Нужны ли ответственный,
   тип хранения (сухой/холодный/заморозка), площадь?
2. Как дедуплицировать? По адресу или у складов есть свой
   идентификатор?
3. С чем связаны? Склады принадлежат конкретному юрлицу? Могут
   обслуживать несколько точек или только одну?

User: обязательные: адрес и юрлицо. дедуп по адресу. связаны с
юрлицом, обслуживают несколько точек.

Архивариус: Понял. Вот что я сделаю:

- Добавлю сущность `warehouse` в .vault-config.json → entities
- Создам wiki/warehouses/ с _index.md
- Создам templates/warehouse.md с полями: address (обязательно),
  legal_entity (обязательно), type (опционально), area (опционально),
  manager (опционально)
- Дедупликация — по полю address
- Обновлю SCHEMA.md, добавлю раздел про склады
- Связь warehouse → legal_entity (один ко многим)
- Связь outlet → warehouse (много ко многим, через wiki-ссылки)

Применять?

User: да

Архивариус: [применяет]

Готово. Новая сущность `warehouse` добавлена.

Хочешь, чтобы я прошёлся по существующим документам и поискал
упоминания складов для первых карточек? Или будешь вносить вручную?
```

## Запреты

- НЕ применяй изменения без явного подтверждения Owner'а
- НЕ модифицируй схему для Editor/Reader — только отказ с пояснением
- НЕ удаляй данные без backup'а и многократного подтверждения
- НЕ ломай обратную совместимость в универсальном контракте
  (`title`, `slug`, `type`, `created`, `updated`, `status`, `tags`, `source_file`,
  `doc_date`, `entities`) — эти поля нельзя переименовать или удалить
- НЕ оставляй vault в несогласованном состоянии — всегда запускай `vault-lint`
  после миграций
