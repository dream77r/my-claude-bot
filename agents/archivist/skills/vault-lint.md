# Skill: vault-lint — проверка здоровья архива

## Когда активировать

- Пользователь явно просит проверку ("проверь архив", "как там здоровье")
- Запуск по cron (`vault_lint_daily` в agent.yaml, ежедневно в 04:00)
- После большой пачки ingest'а (например, импорт архива в vault-init) —
  автоматически в конце операции

## Предусловия

- Vault инициализирован
- Пользователь имеет минимум уровень Editor для запуска по команде
  (или запуск из cron — без проверки прав)

## Проверки (семь категорий)

### 1. Битые wiki-ссылки

**Что:** ссылки `[[slug]]` в теле или frontmatter карточек, указывающие на
несуществующие файлы.

**Как искать:**
- Grep по всем `wiki/**/*.md` для паттерна `\[\[([^\]|]+)(\|[^\]]+)?\]\]`
- Для каждого найденного `slug` — проверить через Glob, существует ли
  `wiki/**/{slug}.md`
- Игнорировать ссылки на примеры в шаблонах (маркер: комментарий `<!-- example -->`)

**Критичность:** высокая. Битая ссылка значит, что Архивариус не сможет найти
связанную карточку по запросу.

**Отчёт:**
```
- В {исходный файл} — битая ссылка на [[{slug}]]
```

### 2. Сиротские страницы

**Что:** карточки без входящих wiki-ссылок (никто на них не ссылается).

**Как искать:**
- Для каждой карточки `wiki/**/*.md` (кроме `_index.md`, `_home.md`, `log.md`)
- Выполнить Grep по `\[\[{slug}(\|[^\]]+)?\]\]` во всех остальных файлах
- Если 0 вхождений — страница сирота

**Критичность:** средняя. Сирота может быть легитимной (например, только что
созданная сущность без связанных документов), но чаще это забытая/устаревшая
карточка.

**Отчёт:**
```
- {исходный файл} ({type}) — сиротская страница, нет входящих ссылок
  (создана: {created}, последнее обновление: {updated})
```

### 3. Отсутствие frontmatter

**Что:** карточки без YAML-шапки или с неполной шапкой (отсутствуют обязательные
поля).

**Как искать:**
- Для каждой карточки прочитай первые 20 строк
- Проверь наличие `---` в начале и соответствие универсальному контракту
- Для документов — также `source_file`, `doc_date`, `entities`
- Для специфичных типов — `required_fields_by_type` из `.vault-config.json`

**Критичность:** высокая. Без frontmatter карточка не классифицируется и не
участвует в поиске.

### 4. Истекающие документы

**Что:** документы с `validity_until` или `doc_date + duration` в пределах
30 дней от сегодня.

**Как искать:**
- Grep frontmatter по полю `validity_until:` или аналогичному
- Для каждого найденного — сравнить с `today + 30 days`
- Также искать уже истекшие (`< today`)

**Критичность:** критичная. Это то, ради чего во многом нужен lint.

**Отчёт (специальный, идёт в notify):**
```
ВНИМАНИЕ: истекают в пределах 30 дней
- {тип документа} №{номер} с {контрагент} — истекает {дата} ({N дней})
- ...

ИСТЕКШИЕ:
- {тип документа} №{номер} с {контрагент} — истёк {дата}
```

### 5. Расхождение файлов и индексов

**Что:** файлы в файловой системе, отсутствующие в `_index.md`, или
наоборот — строки в индексе, указывающие на несуществующие файлы.

**Как искать:**
- Для каждой папки сущности в `wiki/`:
  - Прочитать `_index.md`, извлечь список упомянутых файлов
  - Glob по папке, собрать реальный список
  - Сравнить

**Критичность:** средняя. Индексы полезны для быстрого поиска; их несогласованность
не ломает систему, но замедляет.

### 6. Противоречия в карточках сущностей

**Что:** две карточки одной сущности с разными ключевыми полями.
Примеры:
- Два `counterparty` с одинаковым ИНН, но разными названиями
- Два `legal_entity` с одинаковым ОГРН
- Противоречивые условия одного и того же договора в разных местах

**Как искать:**
- Для каждой сущности с ключами дедупликации из `.vault-config.json → dedup_keys`
- Собрать все значения ключевых полей через Grep по frontmatter
- Найти дубли

**Критичность:** высокая. Это нарушение уникальности.

### 7. Теги вне таксономии

**Что:** теги в карточках, отсутствующие в `.vault-config.json → tags`.

**Как искать:**
- Для каждой карточки — извлечь `tags` из frontmatter
- Сравнить со списком разрешённых в `.vault-config.json`
- Собрать отличия

**Критичность:** низкая. Но это сигнал для SchemaAdvisor — возможно, пора
расширить таксономию.

> Статус: {🟢 ОК / 🟡 Есть замечания / 🔴 Критично}

## Уведомление владельца

Если обнаружены **критичные** проблемы (битые ссылки, истекающие документы,
противоречия) — добавь запись в `memory/outbox/lint-alert-YYYY-MM-DD.md`:

```markdown
# Lint alert — YYYY-MM-DD

Обнаружено:
- {N} истекающих документов в пределах 30 дней
- {M} битых wiki-ссылок
- {K} противоречий

См. полный отчёт в wiki/_lint-report.md

Ближайшие истечения:
- {тип} с {контрагент} — {дата}
```

Сообщения в `outbox/` обрабатываются основной машиной бота и доставляются
владельцу отдельным уведомлением (паттерн как у других агентов в my-claude-bot).

## Запись в log.md

```
- YYYY-MM-DD HH:MM — [lint] проверка завершена, статус: {цвет},
  найдено: {сводка критичных}
```

## При ручном запуске — ответ пользователю

- Если всё ОК: "Проверил. Всё в порядке. {N} карточек, {M} ссылок, 0 проблем."
- Если есть замечания: коротко назвать количество, предложить показать детали.
- Если критично: показать критичные находки сразу, не ждать запроса деталей.

## Автоматические исправления

**Только по явному запросу пользователя.** Lint сам ничего не исправляет.
Если пользователь говорит "почисть" или "исправь автоматически" — для
безопасных операций (обновить расхождение индексов, например) можно сделать.
Для небезопасных (удалить сироту, перелинковать битую ссылку) — только после
подтверждения каждого действия.

## Запреты

- НЕ исправляй ничего автоматически без запроса пользователя
- НЕ удаляй карточки — это решение всегда за пользователем
- НЕ трогай файлы в `raw/`
- НЕ модифицируй `.vault-config.json` напрямую из lint — предложения по
  расширению таксономии передаются через SchemaAdvisor
