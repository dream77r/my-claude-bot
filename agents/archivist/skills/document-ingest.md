# Skill: document-ingest — приём документа

## Когда активировать

- Пользователь прислал файл (PDF/DOCX/изображение)
- Пользователь явно говорит "загрузи", "прими документ", "запиши договор"

## Предусловия

1. **Vault должен быть инициализирован.** Проверь `.vault-config.json →
   initialized: true`. Если нет — переведи разговор на `vault-init` и остановись.
2. **У пользователя должны быть права Editor или Owner.** Проверь его Telegram ID
   против `access.owner` и `access.editors`. Если Reader — вежливый отказ:
   "Загружать могут только владелец и редакторы архива. Я могу найти тебе
   что-то из уже загруженного?"

## Workflow (7 шагов)

### Шаг 1 — Сохранить сырой файл

- Скопировать файл в `memory/raw/` под именем `YYYY-MM-DD_slug.ext`
- Имя: lowercase, дефисы вместо пробелов, без спецсимволов
- `YYYY-MM-DD` — дата документа, если удаётся извлечь из содержимого;
  иначе — дата получения
- Если файл с таким именем уже существует — добавить суффикс (`_2`, `_3`)

### Шаг 2 — Извлечь содержимое

- Прочитай файл (Read умеет PDF и DOCX, встроенный OCR для сканов)
- Извлеки все поля, которые относятся к типу этого документа
- Определи `document_subtype` (`contract`, `act`, `invoice`, ...) из
  `.vault-config.json → document_types`
- Если тип не очевиден — попытайся определить по содержимому (слова "договор",
  "акт сверки", "счёт на оплату"); если и это не помогает — спроси пользователя

### Шаг 3 — Определить связанные сущности

Для каждой сущности из `.vault-config.json → entities`:
- Извлеки из документа идентифицирующие поля (ИНН для организаций,
  ФИО для персон, названия для проектов)
- Проверь `wiki/{entity_type}/` — есть ли карточка с таким идентификатором?
  Используй Glob + Grep по frontmatter
- Дедупликация по ключам из `.vault-config.json → dedup_keys`

**Случай 1:** сущность найдена → использовать существующий slug.
**Случай 2:** сущность не найдена, но все ключевые поля есть → создать карточку
автоматически из шаблона `agents/archivist/templates/{entity_type}.md`, сообщить
пользователю "создал новую карточку для {X}".
**Случай 3:** сущность не найдена и ключевые поля неполные → переспросить
пользователя. **Никогда не создавай сущность с неполным ключом**, это испортит
дедупликацию.

**Обязательное правило:** у документа должна быть минимум одна связь с сущностью.
Если не удалось связать — не создавай карточку документа, переспроси.

### Шаг 4 — Создать карточку документа

- Файл: `memory/wiki/{document_category}/YYYY-MM-DD_subtype_slug.md`
  - `{document_category}` — где хранятся документы этого подтипа. Обычно
    `contracts/` для договоров и `documents/` для всей первички. См. SCHEMA.
- Основа: шаблон `agents/archivist/templates/{document_subtype}.md`
- Заполни все обязательные поля из `required_fields_by_type`
- Frontmatter должен содержать:
  - `source_file: "raw/{filename}"`
  - `doc_date: {YYYY-MM-DD}`
  - `entities: {map}` с wiki-ссылками на все связанные сущности
  - `tags: []` из таксономии
  - `status: active` (или `draft` если не хватает данных)
- В теле — структурированная информация из документа

### Шаг 5 — Обновить карточки связанных сущностей

Для каждой сущности, упомянутой в документе:
- Прочитать её карточку
- В соответствующую секцию ("Договоры", "Документы", "Связи") добавить
  wiki-ссылку на только что созданную карточку документа
- Обновить поле `updated` в frontmatter
- Если в документе появилась новая информация (новый телефон, адрес, контактное
  лицо контрагента) — обновить карточку с пометкой даты источника
- Применить правила разрешения конфликтов из CLAUDE.md → "Политика конфликтов"

### Шаг 6 — Обновить индексы

- `wiki/{document_category}/_index.md` — добавить новую строку (таблица:
  дата | тип | стороны | сумма | срок | ссылка)
- `wiki/{entity_type}/_index.md` — если была создана новая сущность, добавить её
  в индекс
- Если в карточке сущности появились новые теги — добавить в
  `.vault-config.json → tags` (после запроса пользователю или автоматически,
  если тег — просто новая сущность, например `le-{slug}`)

### Шаг 7 — Записать в журнал и ответить

- `wiki/log.md`:
  ```
  - YYYY-MM-DD HH:MM — [ingest] принят {subtype} от {doc_date},
    связан с {entity_list}, {новая/существующая} сущность
  ```
- Ответ пользователю (короткий, без лирики, без технических путей):
  ```
  Записал: {doc_subtype_human} от {doc_date}
  С кем: {primary_counterparty_or_entity}
  Предмет: {short_description}
  Связал с: {entity_list}
  {Если создана новая сущность} Контрагент новый, завёл карточку.
  {Если требуется внимание} Не смог извлечь: {missing_fields}, уточни если важно.
  ```

## Обработка нескольких файлов

Если пользователь кинул несколько файлов подряд:
- Обработай их последовательно
- В конце — **один сводный отчёт**: "Принял N документов: ..."
- Если среди них дубли (по номеру/дате/сторонам) — не создавай вторую карточку,
  сообщи о дубле

## Что делать при ошибках

**Файл нечитаем (битый PDF):**
- Не сохраняй в raw/
- Ответ: "Не смог прочитать файл, кажется он повреждён. Пришли ещё раз?"

**Язык документа не распознан:**
- Сохрани в raw/, попытайся извлечь базовые поля
- Ответ: "Сохранил файл, но текст не смог нормально прочитать. Это {язык}?"

**Тип документа не определяется:**
- Сохрани в raw/, **не создавай карточку**
- Спроси: "Это какой тип документа? {список возможных из document_types}"

**Ключевые поля для дедупликации отсутствуют (нет ИНН у новой организации):**
- Сохрани в raw/, **не создавай карточку сущности**
- Спроси: "У этого контрагента нет ИНН в документе. Это {название}? У тебя
  уже есть карточка, или мне создать новую без ИНН?"

## Обязательные правила

- Файлы в `raw/` **никогда не изменяются**
- Карточки в `wiki/{entity_type}/` **никогда не удаляются** автоматически
- Любое изменение — запись в `log.md`
- Никогда не создавай битые wiki-ссылки
- Никогда не используй теги вне таксономии
- Никогда не придумывай данные, которых нет в документе

## Формат ответа

- Средняя длина, структурированный
- Без технических путей (wiki/..., raw/...)
- Человеческие ссылки ("договор №X от YYYY-MM-DD с {контрагент}")
- Отдельно пометить всё, что требует внимания пользователя
