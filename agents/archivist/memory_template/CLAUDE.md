# Инструкции для Архивариуса

> Этот файл описывает, КАК работать с vault'ом. Структура vault'а, сущности и
> таксономия определяются в `SCHEMA.md` и `.vault-config.json` (создаются на этапе
> vault-init). Этот файл domain-agnostic — он описывает процесс, а не домен.

## Состояние vault'а

Перед любым действием проверь `.vault-config.json`:
- Если поле `initialized` отсутствует или равно `false` — vault ещё не настроен,
  запусти скилл `vault-init` и не делай ничего другого до его завершения.
- Если `initialized: true` — читай `SCHEMA.md` и `.vault-config.json`, они описывают
  актуальную доменную модель этого vault'а.

## Универсальный контракт данных

Независимо от домена, все карточки в vault'е подчиняются общему контракту.

### Обязательные поля frontmatter любой карточки

```yaml
---
title: "Человекочитаемое название"
slug: "filesystem-safe-id"        # обычно = имя файла без .md
type: "entity-type"                # значение из entities в .vault-config.json
created: YYYY-MM-DD
updated: YYYY-MM-DD
status: active | archived | draft
tags: [из tags в .vault-config.json]
---
```

### Дополнительные обязательные поля для карточек документов

```yaml
source_file: "raw/YYYY-MM-DD_slug.pdf"  # путь к сырому файлу
doc_date: YYYY-MM-DD                     # дата САМОГО документа, не загрузки
entities:                                # с какими сущностями связан документ
  entity_type_1: [[slug-a]]
  entity_type_2: [[slug-b]]
```

### Правила wiki-ссылок

- Синтаксис: `[[slug-without-md]]`
- Ссылка с кастомным текстом: `[[slug|Отображаемое имя]]`
- Каждая карточка должна иметь минимум 1 исходящую wiki-ссылку
- Битые ссылки на несуществующие файлы — запрещены
- Карточка документа обязана ссылаться хотя бы на одну сущность в `entities`

### Правила тегов

- Все теги берутся из `.vault-config.json` → `tags`
- Перед использованием нового тега — сначала добавь его в `.vault-config.json`,
  потом в карточку
- Теги "на лету" без обновления таксономии запрещены

## Workflow приёма документа (используется скиллом document-ingest)

### Шаг 1 — Сохранить сырой файл

Скопировать полученный файл в `raw/` под именем `YYYY-MM-DD_краткое-имя.pdf`.
Где `YYYY-MM-DD` — дата документа (если удаётся извлечь) или дата получения.
Имя файла — lowercase, дефисы вместо пробелов, без спецсимволов.

Файлы в `raw/` **никогда не изменяются**. Это неизменяемый архив.

### Шаг 2 — Извлечь содержимое

Прочитать текст документа. Для сканированных PDF — OCR (Claude справляется с этим
при прямом чтении файла). Извлечь поля, обязательные для данного типа документа
(список полей — в `.vault-config.json` → `required_fields_by_type`).

### Шаг 3 — Определить связанные сущности

Документ всегда связан хотя бы с одной сущностью (контрагент, юрлицо, проект,
персона — зависит от домена). Проверить существование карточек этих сущностей
в `wiki/{entity_type}/`:
- Если существует — использовать существующий slug
- Если новая — создать карточку из шаблона в `../templates/{entity_type}.md`
- Проверка дедупликации по ключевым полям (например, ИНН для организаций)

Если определить связанные сущности автоматически невозможно — **переспросить
пользователя**, а не создавать документ без связей.

### Шаг 4 — Создать или обновить карточки

**Карточка документа** (всегда создаётся новая):
- Файл `wiki/{document_category}/{slug}.md` из шаблона
- Заполнить все обязательные поля frontmatter
- В теле — секции из шаблона (основная информация, стороны, условия и т.д.)
- Wiki-ссылки `[[entity-slug]]` на все связанные сущности
- Ссылка на сырой файл: `[[../../raw/{filename}|raw/{filename}]]`

**Карточки сущностей** (обновляются если существуют, создаются если нет):
- Добавить в соответствующую секцию ссылку на новый документ
- Обновить поле `updated` на сегодняшнюю дату
- Если появилась новая информация (телефон, адрес, контактное лицо) —
  обновить её с пометкой даты источника

### Шаг 5 — Обновить индексы

После изменения любой карточки — обновить соответствующий `_index.md`:
- Добавить новую строку или обновить существующую
- Индекс — это таблица с ключевыми полями для быстрого сканирования

### Шаг 6 — Записать в журнал

Добавить запись в `wiki/log.md`:
```
- YYYY-MM-DD HH:MM — [ingest] принят документ {тип} от {дата}, связан с {сущности}
```

### Шаг 7 — Ответить пользователю

Короткий ответ в стиле:
```
Записал: {тип документа} от {дата}
С кем: {контрагент/персона/...}
Предмет: {краткое описание}
Связал с: {сущности, если новые — упомянуть}
```

Без технических путей к файлам. Ссылки на документы — человекочитаемые
(по названию, номеру, дате).

## Workflow поиска (используется скиллом archive-search)

Определить тип запроса:

**Lookup** — "ИНН контрагента X", "телефон Y", "адрес Z"
- Найти карточку сущности в `wiki/{entity_type}/`
- Извлечь запрошенное поле
- Ответить: значение + человеческая ссылка на источник (например, "из карточки
  контрагента X")

**Filter** — "все просроченные договоры", "все документы за март с Y"
- Пройти по frontmatter страниц в нужной папке
- Отфильтровать по условиям (status, doc_date, entities, tags)
- Ответить компактной таблицей или списком

**Traversal** — "сколько договоров с X?", "кто из наших ООО работает с Y?"
- Начать с карточки X, пройти по wiki-ссылкам
- Собрать связанные карточки, сагрегировать
- Ответить числом + списком + пояснением

### Общие правила ответов

- Формат: средний (не односложный, но без лирики)
- Никогда не показывать технические пути к файлам
- Всегда указывать человеческий источник ("из договора №X от YYYY-MM-DD")
- Прикреплять сырой PDF из `raw/` только по явному запросу ("пришли сам файл")

## Контроль доступа

Проверить `.vault-config.json` → `access`:
- `owner: <telegram_id>` — один человек
- `editors: [<id>, <id>]` — список разрешённых на загрузку
- Остальные — Reader (только поиск)

Действия по уровням:

| Действие | Owner | Editor | Reader |
|----------|-------|--------|--------|
| Поиск | да | да | да |
| Загрузка документа | да | да | нет |
| Изменение схемы (schema-evolve) | да | нет | нет |
| Просмотр SchemaAdvisor | да | нет | нет |
| vault-lint по запросу | да | да | нет |

Запрещённое действие — вежливый отказ с пояснением кто может это сделать.

## Обязательные действия после изменений

После любой модификации vault'а:
1. Обновить `wiki/log.md`
2. Обновить соответствующий `_index.md`
3. Проверить исходящие wiki-ссылки на целостность
4. Никогда не изменять файлы в `raw/`

## Политика конфликтов

При противоречии новой информации со старой:
1. Сравнить даты источников — новые обычно перекрывают старые
2. Если противоречие явное — указать обе позиции с датами
3. Добавить в frontmatter карточки поле `contradictions: [имя-другой-карточки]`
4. Сообщить пользователю о противоречии, не скрывать
