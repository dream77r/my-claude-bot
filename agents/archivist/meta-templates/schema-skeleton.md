# Скелет SCHEMA.md

> Этот файл — шаблон для генерации `memory/SCHEMA.md` конкретного vault'а.
> Используется скиллом `vault-init` после согласования с пользователем.
> Плейсхолдеры `{{...}}` заменяются на конкретные значения под домен.

---

```markdown
# Схема {{DOMAIN_NAME}}

## Домен

{{DOMAIN_DESCRIPTION}}

Архив хранит {{WHAT_IS_STORED}}. Отвечает на вопросы: {{TYPICAL_QUERIES}}.

## Сущности

{{ENTITY_LIST}}

Каждая сущность хранится в своей подпапке `wiki/{entity_name}/` и имеет свой
шаблон в `templates/{entity_name}.md`.

## Правила именования файлов

- **Имена файлов:** lowercase, дефисы вместо пробелов, без спецсимволов
- **Сущности:** `{{ENTITY_NAMING_PATTERN}}`
- **Документы:** `{{DATE}}_{{SHORT_SLUG}}.md` (дата + короткое имя)
- **Индексы:** `_index.md` внутри каждой папки сущности
- **Исходники:** `raw/{{DATE}}_{{SLUG}}.pdf` (никогда не изменяются)

## Формат страниц

Каждая страница начинается с YAML frontmatter по универсальному контракту:
- `title`, `slug`, `type`, `created`, `updated`, `status`, `tags` — обязательно
- Для документов дополнительно: `source_file`, `doc_date`, `entities`,
  `document_subtype`

Дополнительные поля определяются в `.vault-config.json` →
`required_fields_by_type`.

## Таксономия тегов

{{TAXONOMY_LIST}}

Новый тег добавляется только через обновление `.vault-config.json` → `tags`.
Теги "на лету" запрещены.

## Типы документов

{{DOCUMENT_TYPES_LIST}}

Каждый подтип имеет свой шаблон в `templates/{document_subtype}.md` и
обязательные поля в `.vault-config.json → required_fields_by_type`.

## Правила wiki-ссылок

- Используй `[[имя-файла-без-md]]` для ссылок внутри vault
- Каждая карточка должна иметь минимум 1 исходящую wiki-ссылку
- Ссылки с кастомным текстом: `[[slug|Отображаемое имя]]`
- Проверяй существование целевого файла — не создавай битые ссылки
- Карточка документа обязана ссылаться на минимум одну сущность в `entities`

## Политика дедупликации

{{DEDUP_RULES}}

## Политика обновления

Когда новая информация конфликтует с существующей:
1. Сравни даты — новые источники обычно перекрывают старые
2. Если есть противоречие — укажи обе позиции с датами и ссылками
3. Отметь `contradictions: [имя-страницы]` в frontmatter
4. Сообщи пользователю о противоречии

## Политика жизненного цикла

- `active` — карточка используется прямо сейчас
- `expiring-soon` — документ истекает в пределах 30 дней (ставится lint'ом)
- `expired` — документ истёк
- `archived` — сущность закрыта/не используется, но хранится для истории
- `draft` — черновик, не финализирован

## Обязательные действия после изменений

ПОСЛЕ любого изменения в vault обязательно:
1. Обновить `wiki/log.md` — добавить запись о действии
2. Обновить соответствующий `_index.md`
3. Проверить кросс-ссылки — обновить страницы, которые ссылаются на изменённую
4. Не изменять файлы в `raw/` — это неизменяемый архив

## Lint-проверки

Периодически (ежедневно по cron или по запросу) проверяй:
- **Битые wiki-ссылки** — ссылки на несуществующие файлы
- **Сиротские страницы** — страницы без входящих ссылок
- **Отсутствие frontmatter** — страницы без YAML-шапки
- **Разрыв индексов** — файлы, отсутствующие в `_index.md`
- **Истекающие документы** — `doc_date + validity < today + 30 дней`
- **Противоречия** — карточки одной сущности с разными ключевыми полями
- **Теги вне таксономии** — теги, отсутствующие в `.vault-config.json`

Результаты — в `wiki/_lint-report.md`.
```

## Правила использования этого скелета

1. `{{DOMAIN_NAME}}` — короткое название домена (например, "Контрактный архив")
2. `{{DOMAIN_DESCRIPTION}}` — 1-2 предложения о том, что это за архив
3. `{{ENTITY_LIST}}` — список сущностей с описаниями, формат:
   ```
   - **entity_name** (`wiki/entity_name/`) — что это, какие обязательные поля
   ```
4. `{{ENTITY_NAMING_PATTERN}}` — конкретный пример имени файла сущности
5. `{{TAXONOMY_LIST}}` — разделы таксономии с тегами, формат:
   ```
   **По {критерий}:**
   - `tag-1` — что означает
   - `tag-2` — что означает
   ```
6. `{{DOCUMENT_TYPES_LIST}}` — подтипы документов с обязательными полями
7. `{{DEDUP_RULES}}` — по каким полям проверять дедупликацию
   (например, "для организаций — по ИНН")
