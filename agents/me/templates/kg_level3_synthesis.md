## Задача

### 1. Найди паттерны
- Какие темы повторяются чаще всего?
- Какие темы набирают momentum (частота растёт)?
- Какие темы затухают (были частыми, стали редкими)?

### 2. Проследи эволюцию решений
- Какие решения принимались? Менялись ли они?
- Есть ли решения, которые откладываются раз за разом?
- Есть ли противоречия между решениями разных дней?

### 3. Выяви кластеры
- Какие сущности всегда упоминаются вместе?
- Есть ли изолированные темы (упоминаются, но ни с чем не связаны)?
- Какие сущности являются "хабами" (связаны со многими)?

### 4. Обнови synthesis-страницы

Используй инструменты для обновления файлов:

**wiki/synthesis/recurring-themes.md** — повторяющиеся темы с трендами:
```markdown
# Повторяющиеся темы

## Активные (растут)
- **Тема** — N упоминаний за период, тренд: рост
  - Связанные: [[сущность1]], [[сущность2]]

## Стабильные
...

## Затухающие
...

Обновлено: YYYY-MM-DD
```

**wiki/synthesis/decision-timeline.md** — хронология ключевых решений:
```markdown
# Хронология решений

## YYYY-MM-DD
- Решение: описание ([[связанные сущности]])
- Статус: принято / отложено / отменено

Обновлено: YYYY-MM-DD
```

**wiki/synthesis/knowledge-clusters.md** — кластеры связанных знаний:
```markdown
# Кластеры знаний

## Кластер: Название
- Хаб: [[главная сущность]]
- Участники: [[сущность1]], [[сущность2]], ...
- Контекст: что связывает эти сущности
- Сила: N (суммарная сила связей)

Обновлено: YYYY-MM-DD
```

### 5. Обнови backlinks

Если в wiki/entities/ или wiki/concepts/ есть страницы упомянутых сущностей,
добавь секцию `## Backlinks` (или обнови существующую):

```markdown
## Backlinks
- [[Связанная сущность]] — контекст (дата)
```

## Правила
- Обновляй существующие файлы через Edit, не через Write
- Если файл не существует — создай через Write
- Не удаляй старую информацию — дополняй
- Обнови index.md если создал новые synthesis-страницы
- Не выдумывай паттерны — только то, что видно в данных

### ⛔ Игнорируй собственную инфраструктуру агента

Никогда не считай темой, паттерном, кластером или решением следующие имена
(это компоненты тебя самого, не знание о мире пользователя):
`SmartTrigger`, `deadline_check`, `morning_briefing`, `evening_digest`,
`news_monitor`, `knowledge_graph`, `kg_level1/2/3`, `heartbeat`, `smart_heartbeat`,
`dream`, `dispatcher`, `bus`, `FleetBus`, `wiki` (как объект), `memory`, `daily`,
`summaries`, `graph.json`, `log.md`, `profile.md`, `agent.yaml`,
`интеграция инструментов`, `интеграция-инструментов`,
`Automated Deadline Management`, `Information Access`, `Notification System`.

Если такие имена встречаются в саммари как «темы» или «решения» — это мусор от
старого бага. Не амплифицируй их в synthesis-страницах. Если в существующих
synthesis-страницах есть упоминания этих имён — оставь их в покое (не удаляй
сам, пользователь чистит вручную), но не добавляй новых.

Триггер «запустился и проверил wiki/дедлайны» — НЕ событие и НЕ решение.
Это работа агента, она не должна отражаться в графе знаний пользователя.

Если после фильтрации значимых паттернов не осталось — верни пустые массивы
и не обновляй synthesis-файлы. Лучше пустой synthesis, чем фальшивый.
