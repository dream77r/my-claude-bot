# Design: GitHub Sync — Клиентские бэкапы

**Статус:** Утверждён, готов к реализации  
**Дата:** 2026-04-14  
**Приоритет:** Phase 5+

---

## Суть

Автоматическая ежедневная синхронизация данных каждого клиента в приватный GitHub репозиторий. Защита от трёх сценариев: технической аварии, галлюцинирующего агента, миграции/переезда.

---

## Understanding Summary

- **Что строим:** Ночной cron-джоб, который пушит данные всех клиентов в один приватный GitHub репо
- **Зачем:** Страховка — новый агент может подхватить работу с любой точки истории
- **Для кого:** Клиенты Telegram-бота (до 100 человек)
- **Что хранится:** Всё — память агентов, история разговоров, конфиги
- **Когда:** Раз в ночь автоматически + по запросу `/backup-now`
- **Восстановление:** Клиент запрашивает ссылку → вставляет новому агенту → агент скачивает и продолжает

## Механизм синхронизации

**Ночной cron-джоб (03:00 UTC):**

```
1. Получить список всех активных клиентов из БД бота
2. Для каждого клиента:
   a. Собрать данные всех агентов (memory/, conversations/, config/)
   b. Скопировать в локальную папку репо
   c. git add + git commit "daily backup: 123456_@nikname [YYYY-MM-DD]"
3. Один git push в конце
4. Записать результат в лог синхронизации
```

**Обработка ошибок:**
- Один клиент упал → пропустить, залогировать, продолжить остальных
- Push не прошёл → повторная попытка через 30 минут → уведомление владельцу

**Новый клиент:**
- При первом запуске бота → автоматически создаётся папка и делается первый коммит

## Команды для клиента

```
/backup-link        — ссылка на свою папку в репо
/backup-now         — принудительный бэкап прямо сейчас
/backup-status      — дата последнего бэкапа, статус
/backup-set-github  — привязать свой личный GitHub (токен + repo-url)
```

---

## Двойная синхронизация (опционально)

Клиент может привязать свой GitHub аккаунт:
- Токен хранится зашифрованно в БД бота
- Синхронизация идёт параллельно: сначала центральный репо, потом личный

---

## Флоу восстановления

```
1. Клиент пишет /backup-link
2. Бот отвечает ссылкой на папку в GitHub
3. Клиент вставляет ссылку новому агенту
4. Новый агент клонирует папку и продолжает с последнего коммита
5. При необходимости — откат на конкретную дату через git history
```

## Decision Log

| Решение | Альтернативы | Почему выбрали |
|---|---|---|
| Один центральный репо | Отдельный репо на клиента | Проще управлять, легко мигрировать потом |
| Папка: `ID_@никнейм` | Только ID | Читаемость при ручном просмотре |
| Ночной cron (03:00 UTC) | Real-time, по запросу | Достаточно для защиты, не нагружает систему |
| Git-история (не перезапись) | Snapshot с перезаписью | Нужен откат при галлюцинирующем агенте |
| PyGitHub / subprocess | GitHub REST API, микросервис | YAGNI — просто, достаточно для 100 клиентов |
| Двойная синхронизация опционально | Только центральный | Гибкость для продвинутых клиентов |

---

## Предположения

- GitHub токен бота хранится в `.env` на сервере
- Объём данных на клиента: десятки МБ максимум
- Клиент не имеет прямого доступа к GitHub (только через команды бота)
- Репо всегда приватный
